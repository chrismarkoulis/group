Τα μέλη της ομάδας συναντήθηκαν στις 8 Ιανουαρίου για να συζητηθούν:

1) Ποιος/ποιοι Controllers θα είναι αρμόδιοι για την δημιουργία των οντοτήτων Customer, Guide

2) Ποιος Controller θα είναι αρμόδιος για την εμφάνιση της αρχικής σελίδας και τι θα δείχνει

3) Ποιες είναι μερικές CRUD operation αρμοδιότητες για τις οντότητες Customer, Guide από τις οντότητες Customer, Guide, Admin

4) Πώς θα διατηρείται η ταυτότητα/id ενός χρήστη καθόλη τη διάρκεια που περιηγείται στην εφαρμογή


Αποφασίστηκαν τα παραπάνω να υλοποιηθούν αντίστοιχα ως εξής:

1) Σχετικά με την δημιουργία οντοτήτων

Θα δημιουργηθεί αρμόδιος Controller με τίτλο RegistrationController, ο οποίος θα ακούει στην διεύθυνση "/register" και θα περιλαμβάνει GetMapping και PostMapping μεθόδους για την δημιουργία Guides και Customers με ελέγχους ορθής εισαγωγής δεδομένων και αντίστοιχης ανακατεύθυνσης - σε περίπτωση επιτυχούς ή όχι δημιουργίας οντοτήτων - σε αντίστοιχες σελίδες

2) Σχετικά με την διαχείρηση της αρχικής σελίδας

Θα δημιουργηθεί ένας Controller με τίτλο HomeController, ο οποίος θα δείχνει - εάν υπάρχουν, όλες τις διαθέσιμες ξεναγήσεις που έχουν δημιουργηθεί, που για αυτές θα υπάρχουν αντίστοιχα πεδία φίλτρων αναζήτησης, με βάση τον ξεναγό (Guide), την τοποθεσία που λαμβάνει μέρος κοκ. Για την αναφερόμενη λειτουργία θα χρησιμοποιείται αντίστοιχα μια Service με τίτλο TourService. Ο λόγος που η σελίδα δείχνει απ ευθείας σε μια λίστα από οντότητες είναι για να ακολουθεί όσο είναι εφικτό το σύγχρονο μοντέλο web εφαρμογών που εμπορεύονται κάποια δραστηριότητα (στην περίπτωση του project παροχή υπηρεσιών ξεναγήσεων).

3) Σχετικά τις αρμοδιότητες λειτουργιών CRUD

Για την οντότητα Guide: θα μπορούν να δουν (Read) τη λίστα όλες οι οντότητες, θα μπορεί να κάνει αλλαγές (Update) μόνο ο ίδιος ο Guide - προφανώς μόνο στον εαυτό του, θα μπορεί να κάνει διαγραφή (Delete) μόνο ο Admin για οποιονδήποτε όπως και ο εκάστοτε Guide για τον εαυτό του. Δημιουργία (Create) ενός Guide θα μπορεί να κάνει κάτα την εγγραφή του (register) ένας οποιοσδήποτε χρήστης (υπενθύμιση: ένα από τα απαραίτητα στοιχεία που θα πρέπει να καταχωρήσει θα είναι ένας έγκυρος κωδικός πιστοποιητικού (Certificate) που θα είναι μοναδικός για κάθε Guide.

Για την οντότητα Customer: θα μπορεί να τους δει (Read) μόνο ο Admin, να κάνει σχετικές διορθώσεις (Update) μόνο ο ίδιος Customer μόνο στον εαυτό του, διαγραφή (Delete) από τον Admin για οποιονδήποτε όπως και ο εκάστοτε Customer για τον εαυτό του. Όπως και για τον Guide, δημιουργία (Create) ενός Customer θα μπορεί να κάνει ένας χρήστης κατά την εγγραφή του (register).

4) Για την διατήρηση της ταυτότητας (id) ενός χρήστη συζητήθηκε εν συντομία να υλοποιηθεί παράδειγμα που έγινε στη διάρκεια των μαθημάτων, χρησιμοποιώντας session μεταβλητή κατά την περιήγησή του. Το παραπάνω θα συζητηθεί περαιτέρω άμεσα.


Miscellaneous: Σχετικά με τους ρόλους των συμβεβλημένων developers για το ομαδικό έργο, παρόλο που όλα τα μέλη έχουν ισότιμο λόγο στην καθολική και επιμέρους δημιουργία του, δηλαδή στα τμήματα Front-end, Back-end και database, ξεχωρίζουν επιπλέον ιδέες και θέσεις ευθύνης για το κομμάτι του Front-end από τον κύριο Χρήστο Μαρκουλή και για το Back-end από τους κυρίους Γεώργιο Παπαϊωάννου και Νικόλαο Σαμπάνη. Αντίστοιχα μοιράζονται σε μεγάλο βαθμό και τα κομμάτια ανάπτυξης κώδικα στο κάθε τμήμα.
